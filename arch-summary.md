# 矿山自动驾驶生产调度平台 · 智能问答 Agent 架构方案摘要

---

## 一、项目背景与目标

矿山自动驾驶生产调度平台已具备地图监控、设备调度、故障查询、任务查询、设备管理、编组管理、运营参数配置等核心能力，但当前以页面操作为主，存在三大痛点：

1. **跨页面信息查找成本高** — 用户需多页面切换核对状态和规则
2. **多接口联合判断成本高** — 同一问题通常依赖多个接口和页面信息
3. **缺少统一编排能力** — 平台有查询/执行接口，但缺少意图识别、权限裁剪、结果回收和交互编排

**建设目标：** 新增一套"单入口、可识别、可查询、可执行、可分发"的智能问答 Agent，负责四类能力：

| 能力 | 说明 |
|------|------|
| 意图识别 | 判断问题类型并路由到正确链路 |
| 查询 | 事实查询、规则说明、页面跳转 |
| 执行 | 用户确认后的受控低风险操作 |
| 分发 | 转发给异常处理 Agent 或数据分析 Agent |

**交互性能硬指标：**

- 单次完整交互总时长 ≤ 30 秒，最长 60 秒
- 任意链路不能连续 > 10 秒无反馈，必须通过 SSE 返回中间状态
- 首期所有执行动作必须用户确认后执行，禁止无确认直执

## 二、范围与边界

### 首期实现

- 单一问答入口 + 统一会话管理
- 对象槽位抽取 + 意图识别 + 上下文补全
- 查询类：页面解释、对象事实查询、规则查询、页面跳转
- 执行类：确认后受控执行 + 高风险操作草案与回跳
- 分发类：转发异常处理 Agent / 数据分析 Agent 并回收结果
- SSE 流式输出、标准结果卡、预设问题、执行确认弹窗、操作留痕
- 接入 7 个页面：地图监控、设备调度、故障查询、任务查询、设备管理、编组管理、运营参数配置

### 首期不实现（需预留扩展）

- 高风险车辆控制、中高风险批量控制、自动调度决策
- 私有化本地大模型 / 向量库 / 日志平台部署
- 异常处理 Agent / 数据分析 Agent 内部实现
- 更多页面深度嵌入

### 扩展原则

- 新动作只需注册工具契约 + 权限映射，不改主链路
- 新 Agent 只需新增路由目标和结果适配器
- 私有化部署只替换底层资源接入层，不改业务协议

## 三、现有平台现状与技术约束

### 可复用能力

- 平台网关（Token 透传、路由、接口访问）
- 业务服务（车辆实时信息、参数配置、编组管理、告警/日志/配置中心）
- 权限体系（组织数据权限 + 页面权限 + 按钮权限）
- 云端数据管理平台（LLM 网关、RAG 服务、Milvus）
- 综合调度平台基础设施（Redis、Nacos、Pg、Kafka）

### 核心约束

- 优先复用现有设施，避免修改核心业务服务内部逻辑
- Agent 必须独立部署，故障不影响平台原有功能
- 当前阶段使用公有云，需现场可访问公网
- 权限控制重点在"能问什么、能看什么、能执行什么"，而非"页面入口是否显示"
- 矿山场景：查询须用热缓存和实时接口，避免旧数据误导；用户职责重叠，按权限约束而非角色分工

## 四、整体架构设计

### 设计原则

1. **单入口** — 用户请求入口只有一个
2. **意图优先** — 先识别意图，再选链路
3. **最小侵入** — 不改现有服务，只新增 Agent 层
4. **权限前置** — 页面/按钮权限决定可问、可看、可执行范围
5. **先完成再优化** — 先做通主链路
6. **复用优先** — LLM/RAG 复用云端平台，缓存/配置/DB 复用调度平台

### 四层架构

```
┌─────────────────────────────────────────────────────────┐
│  接入层：前端页面嵌入入口 → 平台网关 → 权限体系            │
├─────────────────────────────────────────────────────────┤
│  Agent 能力层                                            │
│    ├── 智能问答 Agent（核心，含 5 个子模块）               │
│    ├── 异常处理 Agent（专业域，独立实现）                  │
│    └── 数据分析 Agent（专业域，独立实现）                  │
├─────────────────────────────────────────────────────────┤
│  平台与基础设施层                                         │
│    ├── 现有业务服务（车辆/参数/编组/告警/日志）            │
│    ├── 云端数据管理平台：LLM 网关 / RAG / Milvus          │
│    └── 综合调度平台：Redis / Nacos / Pg / Kafka           │
├─────────────────────────────────────────────────────────┤
│  审计与运维层：问答/调用/确认/执行日志 + 监控告警          │
└─────────────────────────────────────────────────────────┘
```

### 智能问答 Agent 内部五个子模块

| 模块 | 职责 |
|------|------|
| 会话与上下文模块 | session、页面上下文、最近对象/动作/确认状态 |
| 对象槽位模块 | 从问题和上下文提取车辆/编组/参数/时间/动作等结构化字段 |
| 意图识别与路由模块 | 判断查询/执行/异常转发/分析转发 |
| 查询与执行模块 | 工具路由、Tool Gateway 调用、确认校验、草案生成、SSE 中间态 |
| 结果输出模块 | 统一生成结果卡/确认卡/补充信息卡/草案卡，通过 SSE 输出 |

### 请求处理流程

1. 解析会话、页面、对象、权限与历史上下文
2. 抽取对象槽位、动作槽位、时间槽位和范围槽位
3. 识别意图类型（查询/执行/异常转发/分析转发）
4. 槽位不全 → SSE 返回补充信息卡
5. 执行类需确认 → SSE 返回确认卡
6. 查询类 → 直接查询返回结果卡
7. 异常/分析类 → 转发对应 Agent，回收结果统一包装

## 五、核心技术实现

### 5.1 大模型接入

**LLM 三大职责：**
- 意图识别与问题改写
- 对象槽位与动作槽位抽取
- 结果卡文本组织与自然语言表达

**交互控制：**
- 所有模型调用统一经 LLM 网关
- 查询/执行类以结构化工具结果为主，模型只做语言组织，不允许臆造业务结论
- 异常/统计类只展示外部 Agent 已确认结果，不自行推理

**幻觉控制四项规则：**
- 事实查询 → 结论必须来自工具返回值
- 执行判断 → 必须来自规则、权限和可信度校验
- 知识解释 → 必须带引用来源与更新时间
- 异常/统计 → 只展示外部 Agent 已确认结果

### 5.2 问答路由决策树

```
用户问题 → 意图识别
  ├── 异常域        → 异常处理 Agent
  ├── 统计分析域     → 数据分析 Agent
  ├── 执行域        → 工具链路（Tool Gateway）
  ├── 事实查询      → 工具链路（车辆状态/参数值/编组关系/可执行性判断）
  └── 规则/页面说明  → RAG 链路
        └── RAG 失败 → 降级为固定模板
```

**RAG 在线策略：**
- 知识型回答尽量带引用
- 检索失败 → 回退固定模板
- 未命中问题写入数据库，用于知识补齐和高频问题治理
- 日志摘要暂不进在线召回主链路

### 5.3 Tool Gateway（工具网关）

统一代理所有工具调用，核心价值：

- **统一注册**：工具元数据、路由地址、超时/重试/熔断/幂等规则
- **统一鉴权**：权限校验在 Gateway 进行
- **统一回执**：错误码由 Gateway 定义，Agent 映射为结果卡文案
- Agent 侧只覆盖差异化配置（如临时调整超时）

**工具注册规范：**
- 查询工具超时 3000ms，最多快速重试 2 次
- 执行工具超时 5000ms，不自动重试，必须配置幂等规则
- 高风险工具强制 `auditRequired=true`

**示例：**
- `vehicle.realtime.query` → 查询车辆实时状态
- `vehicle.air.off` → 执行空调关闭

### 5.4 对外接口

| 接口 | 说明 |
|------|------|
| `POST /agent/v1/message` | 统一消息入口，返回 requestId，后续通过 SSE 返回 |
| `GET /agent/v1/stream` | SSE 流式返回进度与结果 |
| `POST /agent/v1/confirm` | 用户确认后触发实际执行 |
| `POST /agent/v1/session/reset` | 重置会话 |
| `POST /agent/v1/preset-questions` | 获取预设问题 |

**message 接口：** 除 message 外其他字段可选，未传 sessionId 则默认用 userId。后端从 Redis 补全上下文。

**SSE 事件序列：**
- 确认前：`accepted → processing → need_confirm`
- 确认后：`processing → partial_result（可选）→ completed / failed`
- 补充信息：`need_input`

**confirm 接口幂等策略：**
- `idempotentKey = requestId + actionCode + objectHash + confirmToken`
- 前端确认后立即置灰，后端 Redis 写入 `confirm:{idempotentKey}`，TTL 1 分钟
- confirm 语义 = 确认草案 + 同步触发真实执行
- 执行超时 10 秒，超时返回 failed

### 5.5 对象槽位

把用户自然语言转为结构化参数的中间层：

```json
// "12079 现在能补能吗？"
{
  "objectType": "vehicle",
  "objectId": "12079",
  "actionType": "energy_check",
  "pageCode": "map_monitor"
}
```

**槽位来源优先级：**
1. 用户当前问题明确指定
2. 当前页面明确选中的对象
3. 最近一轮会话中确认过的对象

### 5.6 会话模型与 Redis 设计

- **单用户单会话**：sessionId = userId
- 多页面共用一个会话，页面刷新沿用
- SSE 断线重连恢复原会话和最近事件流
- Redis Key 设计：`agent:pagectx:{sessionId}:{pageCode}`
- TTL 参数统一 Nacos 配置，草案和审计日志持久化在数据库

**页面切换清理规则：**
- 切换页面时不换 sessionId，但必须携带新页面的 pageCode、objectContext
- 对象类型不兼容时清除旧槽位，保留用户级通用上下文
- 用户主动点"新会话"才清除全部上下文

**SSE 重连机制：**
- 支持 Last-Event-ID，eventId = requestId + ":" + seq
- Redis 缓存最近 10 条事件
- 断线重连后补发未收到的事件

### 5.7 权限控制（三层防线）

| 权限类型 | 控制范围 |
|----------|----------|
| 页面权限 | 决定可提问哪些页面域问题、可查看哪些结果 |
| 按钮权限 | 决定可执行哪些命令、可收到哪些确认卡 |
| 组织权限 | 决定可访问和操作哪些组织范围内的对象 |

**权限上下文组装流程：**
1. Token → userId
2. `tr_user_role` → 角色列表
3. `tr_role_permission` + `tb_permission` → 页面/按钮权限
4. `tr_user_organization` → 组织范围
5. Redis 缓存 UserPermissionContext

**裁剪规则：**
- 启动时建立 `queryResource → pagePermission`、`actionCode → buttonPermission` 映射
- 查询链路未命中页面权限 → 直接拒绝
- 执行链路未命中按钮权限 → 直接拒绝，不生成确认卡
- 对象所属组织不在范围内 → 返回"权限不足"
- 裁剪发生在服务端，不依赖前端过滤

### 5.8 数据可信度

在执行或实时查询前校验：时间戳过期、关键字段缺失、状态冲突、权限范围。不同动作使用不同阈值，规则初期在配置表维护，后续下沉规则中心。

### 5.9 内部 Agent 分发协议

**调用模式：** HTTP SSE + WebFlux/Reactor 流式桥接

**请求体示例：**
```json
{
  "requestId": "req_20260325_001",
  "sessionId": "sess_001",
  "question": "最近7天这台车有几次异常停车",
  "context": {
    "pageCode": "map_monitor",
    "objectSlots": { "objectType": "vehicle", "objectId": "12079", "timeRange": "last_7_days" },
    "permissionContext": { "pagePermissions": ["map_monitor:view"], "organizationIds": ["org_a"] }
  }
}
```

**超时阈值：**
- 首次响应超时：5 秒
- 总同步流式超时：20 秒
- 超时 → 降级为失败模板

**降级策略：**
- 异常 Agent 超时 → 提示回故障查询页
- 数据分析 Agent 超时 → 提示缩小时间范围或回原页面
- 返回结构不完整 → 统一降级为失败卡

**结果映射：**
- 外部 Agent 返回 resultPayload → 智能问答 Agent 映射为 answerCard
- 外部 Agent 返回 refs → 统一格式化引用来源
- 外部 Agent 不直接定义前端卡片样式

### 5.10 动作白名单与黑名单

- 首期所有允许动作策略 = `confirm_then_execute`，不存在 `direct_execute`
- 影响车辆安全的动作 → `forbidden` 或至少 `draft_only`
- 具体白名单清单由业务补充，方案先定义表结构和规则

### 5.11 草案数据模型

**存储策略：** Redis（活跃草案）+ Kafka（异步入库）+ 数据库（历史/审计/统计）

**状态机：** CREATED → 后续状态流转

**draftId 规则：** `D + yyyyMMddHHmmss + requestId后8位 + objectHash前8位`

**草案需支持：** 交接班查询历史、追溯采纳率、统计执行转化率

### 5.12 日志与审计

- 所有日志（问答/工具/草案/确认/审计）→ Kafka → 磁盘日志 + 数据库
- 数据库保留结构化索引字段，日志保留 ≤ 30 天
- 日志写入失败不阻断主流程，失败事件异步重试投递 Kafka
- 前端提供点赞/踩反馈，区分 query_feedback 和 execute_feedback
- 踩数据按问题/页面/动作/知识未命中类型聚类，作为优化依据

## 六、页面接入与预设问题

| 页面 | 典型预设问题 |
|------|------------|
| 地图监控页 | 这台车现在在哪里、能补能吗、查看这批车的模式状态 |
| 设备调度页 | 当前调度任务是什么、下一步去哪里、查看调度详情 |
| 故障查询页 | 最近有哪些故障、故障属于什么类型、查看故障详情 |
| 任务查询页 | 今天执行了几个任务、有哪些未完成任务、查看任务详情 |
| 设备管理页 | 设备属于哪个组织、基础信息是什么、打开设备详情 |
| 编组管理页 | 当前编组关系是什么、属于哪个编组、查看编组配置 |
| 运营参数配置页 | 当前补能策略是什么、参数影响什么、切换到参数详情 |

## 八、风险分析与应对

### 技术风险

| 风险 | 应对 |
|------|------|
| 意图识别错误 → 分发到错误链路 | 统一意图分类与分发协议 |
| 对象槽位抽取错误 | "明确优先、模糊追问、默认不执行"规则 |
| 外部 Agent 返回口径不统一 | 统一分发协议 + 结果适配器 |
| 工具参数不标准/权限映射不完整 | 所有工具统一注册契约，强制定义入参/出参/权限/回执 |
| 公有云网络稳定性 | SSE 中间态保障反馈，超时明确降级提示 |

### 业务风险

| 风险 | 应对 |
|------|------|
| 用户期望 Agent 替代异常/分析专业判断 | 明确智能问答 Agent 只负责识别/查询/执行/分发 |
| 页面/按钮权限映射不清 | 明确三类权限分别约束什么 |
| 低风险动作边界不准确 | 首期仅开放明确低风险动作，复杂动作走草案与回跳 |

### 其他风险

| 风险 | 应对 |
|------|------|
| 现场公网波动 | 预留专线或 VPN 方案，关键路径增加重试 |
| 前端接入节奏不一致 | 分批接入，先地图监控页 |
