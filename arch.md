1. 文档概述
1.1 项目背景与目标
矿山自动驾驶生产调度平台已具备地图监控、设备调度、故障查询、任务查询、设备管理、编组管理、运营参数配置、日志与配置等核心能力，但当前仍以页面操作为主，存在以下共性问题：
- 用户需要跨页面查找信息、核对状态、理解规则，学习成本高。
- 同一问题通常依赖多个接口和多个页面信息联合判断，人工切换成本高。
- 现有平台具备查询接口、执行接口、日志与配置能力，但缺少统一的意图识别、权限裁剪、结果回收和交互编排能力。
- 用户希望平台不仅能回答问题，还能帮助完成简单操作、快速定位页面，并把复杂问题自动分发给专业 Agent。
本方案的建设目标是在现有生产调度平台之上新增一套“单入口、可识别、可查询、可执行、可分发”的智能问答 Agent 技术能力。智能问答 Agent 只负责自身四类能力：
- 意图识别
- 查询
- 执行
- 分发
其中：
- 异常诊断类能力转交给 异常处理Agent。
- 统计分析类能力转交给 数据分析Agent。
- 智能问答 Agent 自身不重复建设异常和统计领域的专业逻辑，而是负责识别、转发、回收结果并统一输出。
交互目标如下：
- 单次完整用户交互总时长控制在 30 秒左右，最长不超过 60 秒。
- 任意长耗时链路不能连续超过 10 秒 无反馈，系统必须通过 SSE 流式返回处理中状态、阶段性进度和最终结果。
- 首期所有执行动作均不允许“无确认直执”，必须由用户确认后执行。
1.2 范围与边界
1.2.1 本方案实现范围
本方案聚焦智能问答 Agent 本体，不包含异常处理 Agent 和数据分析 Agent 的内部实现。首期实现范围包括：
- 单一问答入口与统一会话管理。
- 对象槽位抽取、问题意图识别与上下文补全。
- 查询类能力：页面解释、对象事实查询、规则查询、页面跳转。
- 执行类能力：用户确认后的受控执行，以及高风险操作的草案和回跳。
- 分发类能力：根据意图将请求转发到异常处理 Agent 或数据分析 Agent，并统一回收结果。
- SSE 流式输出、标准结果卡、预设问题、执行确认弹窗、操作留痕。
1.2.2 首期不实现范围与扩展思路
以下内容不在首期直接实现范围内，但需要在当前架构中预留扩展能力：
- 高风险单一车辆控制、中高风险批量控制、自动调度决策、复杂跨对象自动执行。
- 私有化环境下的本地大模型、本地向量库、本地日志平台部署。
- 异常处理 Agent 与数据分析 Agent 的专业能力细节。
- 更多页面的深度嵌入接入。
扩展原则如下：
- 当前所有执行动作都通过统一工具契约注册，未来新增动作只需新增工具和权限映射，不需要重构主链路。
- 当前所有外部分发都通过统一分发协议实现，未来新增专业 Agent 时，可直接新增路由目标和结果适配器。
- 当前优先复用云端数据管理平台中的 LLM 网关、RAG 服务、Milvus 能力；未来如果切换为私有化部署，只替换底层资源接入层，不改智能问答 Agent 的业务协议。
1.3 术语与定义
- 智能问答Agent：平台统一智能入口，负责意图识别、查询、执行和分发。
- 异常处理Agent：负责异常域专业能力，包括异常事实查询、异常解释、处置建议和异常结果输出。
- 数据分析Agent：负责统计分析域专业能力，包括统计汇总、趋势分析、指标口径与数据结果输出。
- 对象槽位：问题中与业务对象、动作对象、时间范围、页面上下文相关的结构化参数集合，用于让 Agent 知道“你在问谁、要查什么、要做什么”。
- LLM 网关：统一的大模型接入层，负责鉴权、路由、超时、日志和调用控制。
- RAG：检索增强生成能力，用于规则说明、页面说明、配置说明等知识型问答。
- Function Call：将现有平台接口封装为模型和编排层可调用的标准工具。
- 数据可信度网关：在执行或实时查询前，对时间戳、关键字段、冲突状态和权限范围进行校验的模块。
- 草案：系统为用户生成的待执行方案或待跳转操作说明，本身不直接执行。
2. 现状分析与技术约束
2.1 现有生产调度平台架构
现有生产调度平台已经形成统一入口、统一登录、统一权限和多业务模块协同的基础架构，具备以下现状：
- 已有平台网关，可承接统一 Token 透传、路由和接口访问。
- 已有业务服务支撑车辆实时信息、参数配置、编组管理、告警中心、日志中心、配置中心、分发状态等能力。
- 已有权限体系同时覆盖组织数据权限、页面权限和按钮权限。
- 已有日志、配置记录、告警规则、操作记录、分发记录等可复用能力。
- 已有页面具备承载智能问答嵌入式入口的基础，重点接入页面包括：地图监控页、设备调度页、故障查询页、任务查询页、设备管理页、编组管理页、运营参数配置页。
- 云端数据管理平台已具备 LLM 网关、RAG 服务、Milvus 相关能力，当前方案优先复用该平台能力，不单独新建一套。
- 综合调度平台当前已有 Redis、Nacos、Pg 数据库、Kafka 等基础能力，智能问答 Agent 的会话、确认令牌、配置、持久化和异步日志链路优先复用该能力。
2.2 核心业务流程
2.2.1 调度查询与执行流程
当前平台中的高频操作主要集中在以下几类：
- 查看车辆当前状态、任务状态、位置与风险状态。
- 查看当前参数配置、编组配置及其适用范围。
- 对低风险对象执行简单操作，例如空调开关、白名单模式切换、关注消息开关等。
- 对中高风险对象生成草案并跳回原页面完成后续操作。
智能问答 Agent 在这些流程中的职责是：
- 接收用户问题。
- 识别问题类型、对象槽位与动作槽位。
- 判断是走查询链路、执行链路，还是分发给其他 Agent。
- 在需要时继续向用户追问补充信息或请求确认。
- 返回统一结果卡、确认卡、补充信息卡或草案卡。
2.2.2 异常处理流程
异常处理相关能力不由智能问答 Agent 本体负责设计和实现，统一由 异常处理Agent 提供。智能问答 Agent 在异常处理流程中的职责仅包括：
- 识别问题属于异常处理域。
- 将问题、上下文、对象槽位和权限信息转发给异常处理 Agent。
- 接收异常处理 Agent 返回的结果，并以统一结果卡方式展示给用户。
2.2.3 数据统计与分析流程
统计、汇总、趋势、指标口径类问题不由智能问答 Agent 本体负责设计和实现，统一由 数据分析Agent 提供。智能问答 Agent 在数据分析流程中的职责仅包括：
- 识别问题属于数据分析域。
- 将问题、上下文、对象槽位和权限信息转发给数据分析 Agent。
- 接收数据分析 Agent 返回的结果，并以统一结果卡方式展示给用户。
2.3 技术约束与限制
- 平台需优先复用现有网关、权限、日志和接口，避免修改核心业务服务内部逻辑。
- 智能问答 Agent 必须独立部署、独立运行，故障时不能影响现有平台原始功能。
- 当前阶段优先使用公有云模型、向量库、缓存与日志能力；未来可扩展到私有化部署，但本次不实现。
- 公有云方案要求私有化现场环境可访问公网或建立稳定出网能力。
- 智能问答 Agent 只负责自身能力，异常处理和数据分析能力不在本方案内部重复建设。
- 用户权限控制重点不在“某页面是否显示入口”，而在“当前用户能问哪些问题、能查看哪些结果、能执行哪些命令”。
2.4 矿山场景特殊约束
- 实时场景下，查询链路必须尽可能使用热缓存和实时接口，避免基于旧数据做错误回答。
- 在任何长耗时链路中，不能连续超过 10 秒 无返回，必须给出 SSE 中间事件，例如 accepted、processing、need_confirm、completed。
- 单次完整交互总耗时应控制在 30 秒左右，最长不超过 60 秒，超时后需明确提示用户当前已降级或请求失败。
- 平台用户在实际作业中职责重叠明显，因此不应基于理想化角色分工来设计交互，而应基于权限和对象范围来约束问答与执行。
3. 总体技术方案设计
3.1 设计原则
- 单入口原则：用户请求入口只有一个，统一由智能问答 Agent 接收和分发。
- 意图优先原则：任何请求先识别意图，再选择查询、执行或分发链路。
- 最小侵入原则：不改现有核心业务服务，只新增 Agent 接入层、编排层和知识层。
- 权限前置原则：页面权限和按钮权限决定可问范围、可看结果和可执行命令范围。
- 先完成再优化原则：先做通主链路，再逐步做缓存、并发和性能优化。
- 复用优先原则：模型与知识检索优先复用云端数据管理平台能力，缓存、配置、数据库与消息链路优先复用综合调度平台既有基础设施能力。

3.2.2 架构说明

3.3 模块职责与交互
3.3.1 单一用户请求入口
用户请求入口只有一个，即智能问答 Agent 的统一消息入口。用户只需要在当前页面发起问题，不需要自己判断要调用哪类能力。
智能问答 Agent 收到请求后，执行如下流程：
1. 解析会话、页面、对象、权限与历史上下文。
2. 抽取对象槽位、动作槽位、时间槽位和范围槽位。
3. 识别问题属于查询、执行、异常处理转发、数据分析转发中的哪一类。
4. 若槽位不全，则流式返回补充信息卡。
5. 若是执行问题且需要确认，则流式返回确认卡。
6. 若是查询问题，则直接查询并返回结果卡。
7. 若是异常处理问题，转发到异常处理 Agent。
8. 若是统计分析问题，转发到数据分析 Agent。
9. 回收结果后统一包装为结果卡、确认卡、补充信息卡或草案卡。
3.3.2 模块职责
- 智能问答Agent 说明：负责统一请求入口、会话管理、对象槽位管理、意图识别、查询与执行、权限校验、数据可信度校验、专业 Agent 分发、结果组装、SSE 事件输出、预设问题下发。
智能问答 Agent 内部进一步拆分为五个子模块：
- 会话与上下文模块 说明：保存 session、页面上下文、最近对象、最近动作、最近确认状态。
- 对象槽位模块 说明：从问题和上下文中提取车辆、编组、参数、时间范围、动作对象等结构化字段。
- 意图识别与路由模块 说明：判断当前问题属于查询、执行、异常转发还是分析转发。
- 查询与执行模块 说明：负责工具路由、Tool Gateway 调用、查询结果聚合、确认前校验、确认后同步执行、草案生成、回执映射和 SSE 中间态输出。
- 结果输出模块 说明：统一生成结果卡、确认卡、补充信息卡、草案卡，并通过 SSE 持续输出。
- 异常处理Agent 说明：负责异常域查询、解释、处置建议和结果返回，不在本方案展开内部设计。
- 数据分析Agent 说明：负责统计分析、指标查询、趋势分析和结果返回，不在本方案展开内部设计。
3.3.3 页面接入与预设问题
本期只接入以下页面：
- 地图监控页
- 设备调度页
- 故障查询页
- 任务查询页
- 设备管理页
- 编组管理页
- 运营参数配置页
每个页面可以配置不同的预设问题，示例如下：
- 地图监控页：这台车现在在哪里、这台车现在能补能吗、帮我查看这批车的模式状态
- 设备调度页：这台车当前调度任务是什么、这台车下一步要去哪里、帮我查看调度详情
- 故障查询页：这台车最近有哪些故障、这个故障属于什么类型、帮我查看相关故障详情
- 任务查询页：这台车今天执行了几个任务、当前有哪些未完成任务、帮我查看任务详情
- 设备管理页：这台设备属于哪个组织、设备当前基础信息是什么、帮我打开设备详情
- 编组管理页：当前编组关系是什么、这台车属于哪个编组、帮我查看这个编组配置
- 运营参数配置页：当前补能策略是什么、这个参数影响什么、帮我切换到这个参数详情
3.4 与现有调度平台集成方案
3.4.1 集成方式
推荐采用“页面嵌入 + 网关接入 + Agent 独立服务”的方式：
- 前端仅在指定页面嵌入问答入口与结果卡组件。
- 所有用户请求通过统一消息接口进入智能问答 Agent。
- 智能问答 Agent 通过现有平台网关访问业务接口，不直接嵌入原服务代码。
- 异常处理 Agent 和数据分析 Agent 通过内部服务协议接入，由智能问答 Agent 调度。
- LLM 网关 / RAG / Milvus 统一复用云端数据管理平台能力。
- 会话、确认令牌、可信快照、权限缓存统一复用综合调度平台 Redis，配置项通过 Nacos 管理，草案与日志持久化复用 Pg 和 Kafka。

3.4.3 流转示例说明
以地图监控页提问“12079 现在能补能吗”为例：
1. 页面通过 POST /agent/v1/message 提交最小请求体，至少带 message，并建议带上当前页面上下文。
2. 后端根据 sessionId（默认取当前登录 userId）从 Redis 中补全当前页面、对象、最近查询对象、最近权限上下文。
3. 智能问答 Agent 抽取 vehicle=12079、action=补能判断 等对象槽位。
4. 若槽位不完整，则 SSE 返回 need_input 事件，提示用户补充对象或时间范围。
5. 若槽位完整，则进入查询与执行模块，调用实时工具与可信度校验。
6. 若当前问题属于异常或统计分析，则转发给对应 Agent，并持续通过 SSE 返回进度事件。
7. 若当前结论可信，则输出结果卡；若命中可执行动作，则先输出确认卡，用户确认后才执行。
8. 确认后继续复用原 SSE 通道返回执行中、执行成功或执行失败结果。
3.4.4 权限控制设计
权限设计基于现有表结构与《permission-control-design.md》的“权限前置、多层防线、默认拒绝”原则实现。
现有核心表如下：
- tb_permission：平台权限明细，包含页面权限和按钮权限。
- tb_role：角色定义。
- tb_user：用户定义。
- tr_role_permission：角色与权限的关系。
- tr_user_role：用户与角色的关系。
- tb_organization：组织定义。
- tr_user_organization：用户与组织的关系，决定用户可控制哪些组织的车辆和资源。
后端权限上下文组装流程如下：
1. 从当前登录 Token 或 Session 中拿到 userId。
2. 查询 tr_user_role 获取用户角色列表。
3. 根据角色列表查询 tr_role_permission，再关联 tb_permission 获取页面权限和按钮权限明细。
4. 查询 tr_user_organization 获取用户可控制的组织范围。
5. 在 Redis 中组装并缓存 UserPermissionContext，供当前会话复用。
建议统一权限上下文结构如下：
{"userId": "u1001","roleIds": ["dispatch", "ops"],"pagePermissions": ["map_monitor:view", "parameter_config:view"],"buttonPermissions": ["air_conditioner:execute", "scene_mode:execute"],"organizationIds": ["org_a", "org_b"]}
权限判断规则如下：
- 页面权限 与查询资源绑定，决定用户可以提问哪些页面域问题，以及可以查看哪些查询结果。
- 按钮权限 与执行命令绑定，决定用户可以执行哪些命令，以及可以收到哪些确认卡。
- 组织权限 决定用户可访问和可操作的组织范围内对象。
- 系统启动时建立 queryResource -> pagePermission、actionCode -> buttonPermission 两类映射，运行时转换为 UserPermissionContext 存储并复用。
- 未命中权限的能力默认拒绝，采用代码层硬拦截。
- 页面可见但问答不可见的对象，统一返回“权限不足”。
4. 核心技术实现细节
4.1 大模型接入与交互控制
4.1.1 功能与作用
大模型在本方案中的职责主要有三类：
- 意图识别与问题改写。
- 对象槽位与动作槽位抽取。
- 结果卡文本组织与自然语言表达。
4.1.2 交互控制要求
- 所有模型调用统一经过云端数据管理平台中的 LLM 网关。

- 对事实查询类问题，结论必须来自工具返回值。
- 对执行类问题，执行判断必须来自规则、权限和可信度校验结果。
- 对知识解释类问题，必须带引用来源与更新时间。
- 对异常类和统计类问题，只展示外部 Agent 已确认的结果，不自行推理专业结论。

4.4 工具调用（Function Call）实现
4.4.1 功能说明
Function Call 的作用是把现有平台接口标准化，供智能问答 Agent、异常处理 Agent 和数据分析 Agent 调用。工具层负责：
- 参数校验。
- 统一错误码。
- 回执标准化。
- 与 Tool Gateway 协同。
Tool Gateway 的作用是把分散在平台各服务中的接口，收敛为统一注册、统一鉴权、统一回执的工具访问入口。
可以把它理解成“给 Agent 用的内部接口网关”。
以本项目为例：
- 智能问答 Agent 要查询车辆实时状态，不直接拼接多个服务地址，而是调用 toolCode=vehicle.realtime.query。
- 智能问答 Agent 要执行空调关闭，也不直接调用底层控制服务，而是调用 toolCode=vehicle.air.off。
- Tool Gateway 根据工具注册信息找到真实接口、校验权限、补齐默认超时和默认重试策略，再把结果标准化返回给 Agent。
这种做法的价值在于：
- 多个 Agent 共用同一工具时，不需要各自维护一套接口地址和默认策略。
- 后续接口地址、默认超时、默认重试策略变化时，只改注册中心或 Gateway 配置，不需要逐个改 Agent。
4.4.2 Tool Gateway 责任分工
- 所有工具统一走 Tool Gateway。
- Tool Gateway 或统一注册中心负责维护工具元数据、真实路由地址、默认超时、默认重试、默认熔断、默认幂等规则。
- 权限校验统一在 Tool Gateway 进行。
- Agent 侧只允许覆盖差异化配置，例如某个场景将默认 3000ms 临时提升到 5000ms。
- 工具错误码由 Tool Gateway 统一定义，Agent 负责映射为结果卡可理解文案。

4.12 权限裁剪与脱敏实现方案
本期私有化部署不单独设计脱敏链路，重点实现权限裁剪。
权限裁剪实施方式如下：
1. 在系统启动或权限缓存刷新时，读取 tb_permission / tr_role_permission / tr_user_role / tr_user_organization，构建两类映射：
  - queryResource -> pagePermission
  - actionCode -> buttonPermission
2. 用户请求进入智能问答 Agent 后，先基于当前登录用户组装 UserPermissionContext。
3. 意图识别完成后，若命中查询链路，则根据命中的 queryResource 反查所需页面权限，未命中直接拒绝。
4. 若命中执行链路，则根据 actionCode 反查所需按钮权限，未命中直接拒绝，不生成确认卡。
5. 若对象所属组织不在 organizationIds 中，则直接返回 权限不足。

实现要求如下：
- 权限裁剪发生在智能问答 Agent 服务端，不依赖前端自行过滤。
- 页面可见但问答不可见的对象，统一返回 权限不足。
- 前端不做二次权限推断，只展示服务端返回结果。

- 对于踩的数据，后续可按问题、页面、动作、知识未命中类型进行聚类分析，作为优化依据。
4.13.4 失败策略
- 日志写入失败不阻断主流程。
- 失败事件写入本地错误日志并异步重试投递 Kafka。
4.14 SSE 事件模型与重连机制
4.14.1 事件 ID 规则
建议支持 Last-Event-ID。
规则：
- eventId = requestId + ":" + seq
- 每个 request 独立递增事件序号
4.14.2 最近事件缓存
- Redis 中缓存最近 10 条事件
4.14.3 重连规则
- 前端断线后使用 Last-Event-ID 重连
- 后端根据 requestId 和 Last-Event-ID 补发后续事件
- 可补发事件包括：processing / partial_result / need_input / need_confirm / completed / failed
5. 落地规划与风险应对
5.1 优先级规划
P0
- 智能问答 Agent 单一入口。
- 意图识别、查询、执行、分发四类能力主链路。
- 地图监控页、设备调度页、故障查询页、任务查询页、设备管理页、编组管理页、运营参数配置页接入。
- SSE 流式输出。
- 对外接口与内部 Agent 分发协议。
- 用户确认后执行的受控执行模型。
- 数据可信度网关、草案机制、标准结果卡。
- 权限上下文组装与权限前置校验。
- 首期动作白名单与黑名单表结构。
P1
- 预设问题配置平台化。
- 更多查询工具接入。
- 更丰富的知识来源接入与引用展示。
- 执行工具扩展与更细的对象槽位识别。
P2
- 更细粒度的缓存与并发优化。
- 更复杂的多对象会话管理。
- 二期延伸到后台管理所有查询页面。
- 未来私有化部署切换能力。
5.2 核心风险分析
技术风险
- 意图识别错误导致分发到错误链路。
- 对象槽位抽取错误导致调用对象错误。
- 外部 Agent 返回口径不统一，影响结果卡统一输出。
- 工具调用参数不标准、权限映射不完整，导致执行失败或误拒绝。
- 公有云模型、向量库和外部调用链路的网络稳定性影响响应时长。
业务风险
- 用户期望智能问答 Agent 直接替代异常处理 Agent 或数据分析 Agent 的专业判断。
- 页面权限与按钮权限的映射如果定义不清，会造成“能问不能做”或“应拦未拦”的争议。
- 低风险动作边界定义不准确，会影响平台易用性或安全性。
其它风险
- 现场公网链路波动会影响公有云调用稳定性。
- 页面接入与现有前端节奏不一致，可能影响首期上线范围。
5.3 风险规避与应急措施
针对技术风险
- 统一意图分类与分发协议，避免不同 Agent 自定义返回格式。
- 所有工具统一注册工具契约，强制定义入参、出参、权限和回执结构。
- 对关键链路增加 SSE 中间态返回，避免长时间无反馈。
- 对执行链路强制进行权限校验、可信度校验和用户确认。
- 对对象槽位设置“明确优先、模糊追问、默认不执行”的规则。
针对业务风险
- 明确智能问答 Agent 本体只负责识别、查询、执行和分发，不吞并异常处理 Agent 和数据分析 Agent 的职责。
- 明确页面权限、按钮权限、数据权限分别约束什么问题、什么结果和什么执行命令。
- 首期仅开放明确低风险动作，复杂动作统一走草案与回跳。
针对其它风险
- 公有云调用失败时，优先返回降级结果或提示用户回到原页面。
- 外部 Agent 不可用时，智能问答 Agent 返回“当前专业能力暂不可用”的标准提示，不影响其他功能。
