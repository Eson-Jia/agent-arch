# 智能问答 Agent 架构方案 · 概念词典

---

## 一、角色与系统

### 智能问答 Agent
平台统一智能入口，是整个方案的核心。负责四件事：意图识别、查询、执行、分发。它不处理异常诊断和数据分析的专业逻辑，只负责识别用户意图后把请求路由到正确的链路或转发给专业 Agent，最后统一回收结果输出给用户。

### 异常处理 Agent
负责异常域的专业能力。当用户问"这个异常为什么还没关闭"、"这台车最近有哪些故障"等异常相关问题时，智能问答 Agent 会把请求转发给它。它负责异常事实查询、异常解释、处置建议和结果输出。本方案不设计它的内部实现。

### 数据分析 Agent
负责统计分析域的专业能力。当用户问"最近 7 天这台车有几次异常停车"、"本月产量趋势"等统计类问题时，智能问答 Agent 会把请求转发给它。它负责统计汇总、趋势分析、指标口径与数据结果输出。本方案不设计它的内部实现。

### 云端数据管理平台
已有的外部平台，提供 LLM 网关、RAG 服务、Milvus 向量库等 AI 基础能力。本方案优先复用该平台能力，不单独新建。

### 综合调度平台
已有的基础设施平台，提供 Redis、Nacos、Pg 数据库、Kafka 等基础能力。本方案的会话存储、配置管理、数据持久化、异步日志都复用该平台。

## 二、AI / 大模型相关

### LLM 网关
统一的大模型接入层。所有对大模型的调用都必须经过它，它负责鉴权、路由（选择哪个模型）、超时控制、调用日志和流量管控。可以理解为"调用大模型的统一入口"。

### RAG（检索增强生成）
Retrieval-Augmented Generation 的缩写。当用户问的是规则说明、页面说明、参数含义、操作说明等知识型问题时，系统先从知识库中检索相关文档片段，再把检索结果喂给大模型生成回答。这样模型的回答有据可查，减少编造内容。

### Milvus
开源向量数据库。RAG 的底层存储引擎，把文档切片后转换成向量存储在 Milvus 中，查询时通过向量相似度检索最相关的文档片段。

### Function Call（函数调用 / 工具调用）
让大模型能够调用外部工具的机制。大模型不直接访问数据库或业务接口，而是告诉系统"我需要调用 vehicle.realtime.query 查询车辆 12079 的实时状态"，然后由系统执行实际调用并把结果返回给模型。本方案中，Function Call 把现有平台接口封装为模型可调用的标准工具。

### 幻觉控制
防止大模型编造不存在的事实。本方案的策略是：事实查询必须来自工具返回值，执行判断必须来自规则和校验结果，知识解释必须带引用来源，异常和统计结论只展示外部 Agent 已确认的结果。模型只做语言组织，不允许臆造业务结论。

### 意图识别
判断用户问题属于哪个类型的过程。本方案把意图分为四大类：查询（"这台车在哪"）、执行（"帮我关空调"）、异常转发（"这个故障怎么回事"）、分析转发（"最近 7 天异常统计"）。意图识别的结果决定后续走哪条链路。

## 三、对象槽位与上下文

### 对象槽位（Slot）
把用户自然语言中的关键信息提取成结构化参数的中间层。用户说的是自然语言，Agent 需要的是结构化参数，"对象槽位"就是这个转换层。

**示例：** 用户说"12079 现在能补能吗？"，系统抽取的槽位是：
```json
{
  "objectType": "vehicle",
  "objectId": "12079",
  "actionType": "energy_check",
  "pageCode": "map_monitor"
}
```

槽位包含：对象类型（objectType）、对象 ID（objectId）、动作类型（actionType）、时间范围（timeRange）、页面编码（pageCode）、路由目标（routeTarget）等。

### 槽位来源优先级
槽位可以从三个地方获取，优先级从高到低：
1. 用户当前问题明确指定的对象（"12079 能补能吗"中的 12079）
2. 当前页面明确选中的对象（用户正在查看某台车的详情页）
3. 最近一轮会话中确认过的对象（上一轮问过 12079，这轮直接说"它能补能吗"）

### 上下文补全
当用户的问题信息不完整时，系统从页面上下文和历史会话中补全缺失信息的过程。例如用户在地图监控页选中了车辆 12079，然后问"能补能吗"，系统会自动补全 objectId=12079。

## 四、工具与网关

### Tool Gateway（工具网关）
给 Agent 用的内部接口网关。把分散在平台各服务中的接口收敛为一个统一入口，Agent 不直接调用各个服务的地址，而是通过 toolCode（如 `vehicle.realtime.query`）调用 Tool Gateway，由 Gateway 找到真实接口、校验权限、执行调用并返回标准化结果。

**核心价值：**
- 多个 Agent 共用同一工具时，不需要各自维护接口地址
- 接口地址/超时/重试策略变化时，只改 Gateway 配置
- 统一进行权限校验、熔断、幂等控制

### 工具契约 / 工具注册规范
每个工具在注册到 Tool Gateway 时必须声明的元数据：toolCode（工具编码）、输入参数、输出参数、默认超时、默认重试策略、幂等规则、是否需要审计等。这是一套标准化的"接口说明书"。

### 熔断
当某个工具/服务连续失败达到阈值时，自动停止调用一段时间，避免雪崩效应。Tool Gateway 层统一管理熔断策略。

### 幂等（Idempotent）
同一个请求重复执行多次，结果和只执行一次相同。本方案中执行类工具必须配置幂等规则，确认接口的幂等键 = requestId + actionCode + objectHash + confirmToken，防止用户重复点击确认导致重复执行。

## 五、交互模型与卡片

### SSE（Server-Sent Events）
服务端单向推送事件的技术。服务端通过一个持久的 HTTP 连接，持续向前端推送事件。本方案中所有的中间状态、进度、结果都通过 SSE 推送，而不是前端轮询。

### SSE 事件类型
本方案定义了以下事件类型：
| 事件 | 含义 |
|------|------|
| `accepted` | 请求已接收 |
| `processing` | 正在处理中 |
| `need_input` | 需要用户补充信息（槽位不全） |
| `need_confirm` | 需要用户确认后才能执行 |
| `partial_result` | 部分结果（可选的中间输出） |
| `completed` | 处理完成，返回最终结果 |
| `failed` | 处理失败 |

### Last-Event-ID
SSE 的断线重连机制。前端断线后带上最后收到的事件 ID 重新连接，后端从 Redis 缓存中补发未收到的事件。事件 ID 格式为 `requestId:seq`（请求 ID + 序号）。

### 结果卡（answerCard）
Agent 返回给前端的标准化展示单元。本方案定义了六类卡片：
- **查询结果卡** — 展示查询到的事实信息（如"12079 当前状态：任务执行中"）
- **确认执行卡** — 展示待确认的操作（如"确认关闭 12079 空调？"+ 确认按钮）
- **补充信息卡** — 提示用户补充缺失信息（如"请输入车辆编号"）
- **草案卡** — 展示系统生成的待执行方案，用户可选择采纳或拒绝
- **失败卡** — 展示错误信息和建议操作
- **分发结果卡** — 展示从外部 Agent 回收的结果

### 预设问题
每个页面预先配置的常见问题列表。用户进入页面后可以直接点击预设问题发起提问，降低使用门槛。例如地图监控页预设"这台车现在在哪里"、"这台车能补能吗"等。

## 六、执行与安全

### confirm_then_execute（确认后执行）
首期唯一的执行策略。所有可执行动作必须先生成确认卡，用户明确点击确认后才执行。不存在 direct_execute（直接执行）。

### 确认令牌（confirmToken）
系统为每个待确认动作生成的一次性令牌。包含在确认卡中返回给前端，用户点确认时必须带上这个令牌。令牌有过期时间（TTL），过期后需要重新发起请求。防止未经授权或过时的操作被执行。

### 草案（Draft）
系统为用户生成的待执行方案或待跳转操作说明，本身不直接执行。中高风险操作不直接在 Agent 中执行，而是生成草案，用户可以跳转到原页面查看和确认。

**草案状态机：**
```
CREATED → WAIT_CONFIRM → CONFIRMED → EXECUTING → EXECUTED_SUCCESS / EXECUTED_FAILED
                       → EXPIRED / CANCELLED / REJECTED
```

### draft_only（仅生成草案）
动作策略之一。影响车辆安全的高风险操作使用此策略，Agent 只生成草案和操作说明，不执行，用户需要跳转到原页面手动完成。

### forbidden（禁止执行）
动作策略之一。最高风险等级的操作，Agent 完全不允许触发，即使生成草案也不行。

### 动作白名单 / 黑名单
控制哪些操作允许通过 Agent 执行。白名单中的动作可以执行（首期统一为 confirm_then_execute），黑名单中的动作被禁止。影响车辆安全的动作归入黑名单或至少 draft_only。

### 数据可信度网关
在执行或实时查询前进行数据校验的模块。检查项包括：
- **时间戳校验** — 数据是否过期（如车辆位置数据超过 N 秒未更新）
- **必填字段校验** — 关键字段是否缺失
- **冲突状态校验** — 当前状态是否允许执行该操作（如车辆正在作业中不允许切换模式）
- **权限范围校验** — 操作对象是否在用户权限范围内

不同动作使用不同的可信度阈值。

## 七、权限体系

### 权限前置
先校验权限，再执行后续逻辑。不是"执行完再检查有没有权限"，而是"没有权限就直接拒绝，连确认卡都不生成"。

### 三层权限

| 权限类型 | 作用 | 示例 |
|----------|------|------|
| **页面权限** | 决定用户能问哪些页面域的问题、能看哪些查询结果 | `map_monitor:view` |
| **按钮权限** | 决定用户能执行哪些命令、能收到哪些确认卡 | `air_conditioner:execute` |
| **组织权限** | 决定用户能访问哪些组织范围内的车辆和资源 | `org_a`, `org_b` |

### UserPermissionContext（用户权限上下文）
系统为每个登录用户组装的权限信息集合，缓存在 Redis 中。包含 userId、角色列表、页面权限列表、按钮权限列表、可控组织列表。每次 Agent 处理请求时从缓存读取，决定该用户能问什么、能看什么、能执行什么。

### 权限裁剪
在服务端根据用户权限过滤结果的过程。不依赖前端自行过滤。页面上可见但用户没有问答权限的对象，统一返回"权限不足"。

### queryResource → pagePermission 映射
查询资源与页面权限的对应关系。例如查询"车辆实时状态"需要 `map_monitor:view` 页面权限。系统启动时建立这个映射表，运行时用于判断用户是否有权限查询某个资源。

### actionCode → buttonPermission 映射
执行命令与按钮权限的对应关系。例如执行"关闭空调"需要 `air_conditioner:execute` 按钮权限。未命中则直接拒绝，不生成确认卡。

## 八、会话与存储

### Session（会话）
本方案按"同一用户一个会话"设计，sessionId 默认等于 userId。同一用户打开多个页面共用一个会话，页面刷新沿用原会话。用户主动点"新会话"才清除全部上下文。

### 页面上下文（pageContext）
当前页面的状态信息，包括 pageCode（页面编码）、pageName（页面名称）、objectContext（当前选中的对象）、pageFilters（页面筛选条件）。存储在 Redis 中，key 为 `agent:pagectx:{sessionId}:{pageCode}`。

### requestId
每次用户提问生成的唯一请求标识。SSE 事件、确认令牌、幂等键、日志追踪都以 requestId 为线索串联。

### 可信快照
在执行或查询时，把当前的关键数据状态（如车辆位置、参数值）保存一份快照，用于后续校验数据是否在操作过程中发生了变化。

## 九、分发协议

### 内部 Agent 分发协议
智能问答 Agent 与异常处理 Agent、数据分析 Agent 之间的通信规范。采用 HTTP SSE + WebFlux/Reactor 流式桥接方式，智能问答 Agent 作为调用方建立 SSE 连接，边接收外部 Agent 的中间态事件，边转发给前端。

### resultPayload
外部 Agent 返回的业务结果数据。智能问答 Agent 负责把它映射为前端可展示的 answerCard，外部 Agent 不直接定义前端卡片样式。

### refs（引用来源）
外部 Agent 返回的数据来源引用。智能问答 Agent 统一格式化后展示给用户，让用户知道答案的依据是什么。

### 降级策略
外部 Agent 不可用时的应对方案：
- 异常处理 Agent 超时 → 提示用户回到故障查询页
- 数据分析 Agent 超时 → 提示用户缩小时间范围或回原页面
- 返回结构不完整 → 统一显示失败卡，不透出原始数据

## 十、基础设施

### Redis
内存数据库，本方案中用于：会话上下文存储、权限上下文缓存、确认令牌管理、草案活跃状态、SSE 事件缓存（最近 10 条）、幂等键校验。

### Nacos
配置中心和服务发现。本方案中所有 TTL 值、超时参数、工具注册配置都通过 Nacos 管理，支持运行时动态调整。

### Kafka
消息队列。本方案中用作异步日志通道：问答日志、工具日志、草案变更事件、确认日志、审计日志先写入 Kafka，再异步落库。日志写入失败不阻断主流程。

### Pg（PostgreSQL）
关系型数据库。用于持久化存储：草案历史记录（tb_agent_draft）、审计日志、权限表（tb_permission 等）。支持交接班查询、审计追溯、统计分析。

### TTL（Time To Live）
数据的存活时间。Redis 中的各类 key 都有 TTL，过期自动删除。例如确认令牌 TTL 推荐 1 分钟，会话上下文 TTL 根据业务配置。所有 TTL 参数统一通过 Nacos 管理。

## 十一、设计原则

| 原则 | 含义 |
|------|------|
| **单入口** | 用户只需要在一个入口提问，不需要自己判断该找哪个系统 |
| **意图优先** | 任何请求先判断用户想干什么，再决定走哪条链路 |
| **最小侵入** | 不改现有业务服务代码，只新增 Agent 层 |
| **权限前置** | 先验权限再做事，没权限连确认卡都不生成 |
| **先完成再优化** | 先跑通主链路，缓存和性能优化后面再做 |
| **复用优先** | 能用现有的 Redis/Nacos/Kafka/LLM 网关就不新建 |
| **默认拒绝** | 未明确授权的操作一律拒绝 |
| **明确优先、模糊追问、默认不执行** | 槽位抽取规则：能确定就确定，不确定就追问用户，绝不猜测执行 |
