# 智能问答 Agent 架构方案 · 热点问答集

---

## Q1：基于 Spring AI Alibaba 框架实现会有什么问题？

### 匹配良好的部分

| 方案需求 | 框架支持 |
|----------|---------|
| LLM 调用 | ChatClient + 通义模型原生支持 |
| Function Call | `@Description` + `Function<Req, Resp>` 注册 |
| RAG 基础能力 | DocumentReader / VectorStore / Milvus 集成 |
| SSE 流式输出 | 基于 WebFlux 的 `Flux<ChatResponse>` |
| Prompt 模板 | PromptTemplate + SystemPromptTemplate |

### 核心问题

**问题 1：多 Agent 编排能力不足（高风险）**

框架没有原生多 Agent 协作框架，本质是"单 Agent + 多工具"模型。Agent 间的协议抽象（请求/响应 schema、中间态事件、路由层）全部需要自建。

建议：自建轻量 Agent 编排层 `AgentRouter → AgentClient(HTTP SSE) → AgentResponseAdapter → 前端 SSE`。

**问题 2：SSE 事件模型不匹配（高风险）**

框架的 `stream()` 返回纯文本 token 流，但方案需要在同一 SSE 通道中混合推送业务状态事件 + 确认卡 JSON + 结果卡 JSON。框架没有 eventId 管理、事件缓存、断线补发能力。确认卡需要"暂停"流等待用户确认，与 `Flux<ChatResponse>` 单向推送模型冲突。

建议：不使用 Spring AI 的 stream 做前端 SSE，自建 `SseEmitter` 事件通道。Spring AI 的 stream 只在内部用于接收 LLM 流式返回。

**问题 3：Function Call 与 Tool Gateway 职责冲突（中风险）**

框架的 Function Call 是框架自动触发的，在 `ChatClient.call()` 内部完成，无法在"LLM 决定调工具"和"实际执行"之间插入 Tool Gateway 的鉴权、权限校验、可信度校验、幂等检查。且框架的 Function 注册是代码级的，方案要求运行时动态注册（Nacos）。执行类工具需要暂停等用户确认，但框架是自动同步执行的。

建议：不使用框架自动 Function Call，LLM 返回 tool_call 意图后由自己的编排层拦截，走 Tool Gateway 执行，再把结果手动拼回上下文。

**问题 4：会话上下文管理局限（中风险）**

框架的 `ChatMemory` 管理的是 Message 列表，不是业务上下文（槽位、权限、页面状态）。没有按业务规则选择性清理上下文的能力，不支持多级 key 设计。

建议：ChatMemory 只用于管理对话消息历史，业务上下文自建 Redis 管理层。

**问题 5：模型绑定（低风险）**

框架默认绑定通义模型。如果云端 LLM 网关是私有协议，可能需要自定义 ChatModel 实现。

### 总结

框架承担约 30% 工作量（LLM 调用封装 + RAG 基础链路 + Prompt 工程），剩余 70% 核心业务编排需要自建：SSE 事件编排器、Agent 路由与分发层、Tool Gateway、确认暂停/草案状态机、业务上下文管理、意图识别前置规则引擎。

## Q2：多 Agent 设计有什么问题？

### 架构层面

**问题 1：智能问答 Agent 职责过重，是单点瓶颈**

它同时承担意图识别、槽位抽取、权限校验、可信度校验、工具调用、SSE 编排、结果组装、Agent 分发。本质上是一个编排引擎伪装成 Agent。它挂了全部能力不可用，未来新增 Agent 时所有路由、适配、降级逻辑都堆在它身上。

建议：拆出独立编排层 Orchestrator（确定性代码，不依赖 LLM），真正的"问答 Agent"与异常/分析 Agent 平级。

**问题 2：意图识别与 Agent 路由耦合太紧**

一次 LLM 误判直接打到错误 Agent，没有回退机制。边界模糊的问题很多："12079 为什么停了"可能是异常、也可能是正常调度结束。

建议：意图识别加置信度，低置信度时追问；提供意图纠正机制；高频模糊场景用规则硬编码。

**问题 3：外部 Agent 协议是空壳**

异常处理/数据分析 Agent "不在本方案设计范围内"，但主链路强依赖它们的输入/输出协议。协议只有 JSON 示例没有 schema 约束，三个团队各自实现时联调必然出问题。

建议：即使不设计内部实现，也必须锁定 JSON Schema + 错误码注册表 + Mock 服务。

### 运行时层面

**问题 4：SSE 跨 Agent 透传复杂度被低估**

前端一条 SSE → 智能问答 Agent → 内部再建一条 SSE → 外部 Agent。两条 SSE 桥接涉及断线处理、事件类型过滤/转换、流式 resultPayload 分片到达时的字段映射、eventId 连续性。这是一个完整的流式网关问题。

**问题 5：延迟叠加容易超标**

异常问题链路：LLM 意图识别(2-5s) + 槽位/权限(~1s) + 转发(~200ms) + 异常 Agent 内部处理(?s) + 结果映射(~500ms) + LLM 生成结果卡(2-5s)。如果异常 Agent 内部也调 LLM，整条链路两次 LLM 调用，30 秒非常紧张。

建议：明确各段性能预算。智能问答 Agent 自身 10s，外部 Agent 15s，缓冲 5s。

**问题 6：Agent 间没有共享上下文机制**

多轮对话中，第一轮问异常 Agent，第二轮问分析 Agent，前一轮的结论无法带过去。两个 Agent 结果冲突时没有仲裁机制。

## Q3：如何进行意图识别？

### 三级识别架构

```
用户问题
  │
  ▼
[第一级] 规则引擎（< 50ms）
  │  命中 → 直接路由
  │  未命中 ↓
  ▼
[第二级] LLM 意图识别 + 槽位抽取（2-5s）
  │  高置信度 → 路由
  │  低置信度 ↓
  ▼
[第三级] 追问用户确认
```

### 第一级：规则引擎

对高频确定性问题用关键词 + 正则 + 页面上下文直接判定：

- **预设问题精确匹配** — 用户点击预设按钮，直接使用预定义意图+槽位
- **执行类关键词** — "帮我(关|开|切换|设置|修改)" + 动作对象 → EXECUTE
- **异常类强信号词** — "故障/异常/报警/告警/为什么停了" → EXCEPTION_FORWARD
- **统计类强信号词** — "(最近/过去)...(几次/统计/趋势/汇总)" → ANALYSIS_FORWARD
- **页面跳转类** — "(打开|跳转|查看)...(详情|页面)" → QUERY_NAVIGATE
- **知识类** — "(是什么意思|规则是什么|策略是什么)" → QUERY_RAG

规则层只处理高确定性问题，首期目标 60-70% 规则命中。

### 第二级：LLM 意图识别 + 槽位抽取

一次 LLM 调用同时完成意图分类和槽位抽取，输出结构化 JSON：

```json
{
  "intentType": "QUERY_FACT",
  "confidence": "HIGH",
  "objectSlots": {
    "objectType": "vehicle",
    "objectId": "12079",
    "actionType": "energy_check",
    "timeRange": null
  },
  "reasoning": "用户询问车辆能否补能，属于可执行性判断类事实查询"
}
```

关键设计点：
- 强制 JSON 输出（JSON Mode / BeanOutputConverter）
- 必须带 confidence（LOW 时走追问）
- 带 reasoning（用于日志审计和 badcase 分析）
- 注入页面上下文（消歧关键手段）
- 槽位允许 null，不确定的不填

### 第三级：追问用户

confidence 低于 0.7 或意图为 UNCLEAR 时，返回 `need_input` 事件，给出 2-3 个候选理解让用户选择。

### 模糊场景处理策略

| 用户问题 | 处理策略 |
|----------|---------|
| "12079 为什么停了" | 先查事实状态，异常停车→转异常 Agent，正常完成→返回结果卡 |
| "最近故障多不多" | 含统计语义"多不多" → 走分析 Agent |
| "补能策略有问题吗" | 含诊断语义"有问题吗" → 走异常 Agent |
| "帮我看看这台车" | 默认事实查询，结果卡附带跳转链接 |

**核心原则：页面上下文是消歧第一手段。** 同一句话在不同页面意图可能完全不同。

### 工程建议

- 规则库放 Nacos，运维可热更新
- LLM 调用硬限 5 秒超时，超时降级为规则兜底
- 记录每次识别日志（question、规则/LLM 命中、intentType、confidence）
- 定期分析 UNCLEAR 和 LOW 数据，高频问题补充规则
- 结果卡底部加"这不是我想问的"按钮，触发重新路由

## Q4：Milvus 在项目中的作用及与 Embedding 的配合

### Milvus 的作用

Milvus 只服务于 RAG 链路。当用户问规则说明、页面说明、参数含义等知识型问题时，从 Milvus 检索相关文档片段喂给 LLM 生成回答。事实查询、执行、异常转发、分析转发都不经过 Milvus。

### Embedding + Milvus 工作原理

**离线阶段（知识入库）：**

```
知识文档 → 文档切片(200-500字/chunk) → Embedding 模型 → 向量 → 存入 Milvus
```

Milvus 中每条记录包含：向量（检索用）、原文（回答用）、来源（引用用）、更新时间（时效用）、分类（过滤用）。

**在线阶段（用户提问）：**

```
用户问题 → Embedding 模型 → 问题向量 → Milvus 相似度检索 → Top-K 文档片段
→ 拼装 Prompt（文档片段 + 用户问题）→ LLM 生成回答（带引用来源）
```

### 为什么需要 Embedding 而非关键词搜索

Embedding 理解语义而非匹配字面。用户说"充电规则"，文档写"补能策略"，关键词搜不到，但向量检索能找到，因为语义向量接近。

### 应入库的知识类型

参数规则说明、页面操作说明、告警/故障类型说明、运营参数含义、常见问题 FAQ。

**不应入库：** 车辆实时状态、具体参数值、异常事件记录、统计数据（这些走工具链路或专业 Agent）。

### RAG 降级策略

- 检索到相关文档（相似度 > 阈值） → 生成回答（带引用）
- 相似度低 → 回答 + 标注"信息可能不够准确"
- 完全未检索到 → 返回"未检索到知识"固定模板 + 写入未命中记录表

## Q5：部署后切换 Embedding 模型会导致什么问题？

### 核心问题

**新模型产生的向量和旧模型产生的向量不在同一个语义空间，所有检索全部失效。**

不同 Embedding 模型对同一文本输出完全不同的向量：维度可能不同（Milvus 直接报错）、即使维度相同数值也完全不同（相似度计算失效）。

### 三种表现

| 情况 | 表现 | 后果 |
|------|------|------|
| 维度不同 | Milvus 查询报错 | RAG 100% 不可用 |
| 维度相同但空间不同 | 检索到不相关文档 | 答非所问（最危险，不报错但错误） |
| 维度相同但空间不同 | 所有相似度低于阈值 | 全部降级为"未检索到知识" |

影响范围仅限 RAG 链路（知识型问答），事实查询/执行/异常/分析不受影响。

### 正确的切换方式

```
1. 准备：Milvus 中创建新 Collection（knowledge_v3），保留旧的（knowledge_v2）
2. 重建：用新模型对所有知识文档重新向量化，写入新 Collection
3. 验证：准备测试集，对比新旧检索结果，确认召回率不降
4. 切换：灰度切换（10% → 100%），或低峰期停机切换
5. 清理：确认无问题后删除旧 Collection
```

### 工程建议

- Collection 带版本号（knowledge_v2、knowledge_v3）
- 每条向量的元数据记录 Embedding 模型名
- Nacos 配置 Collection 名和模型名，切换改配置不改代码
- 提前准备全量重建自动化脚本
- 准备 50-100 个标准问题的 RAG 测试集，每次切换跑回归
- 切换窗口安排在低峰期
